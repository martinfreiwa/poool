## Summary

The Admin Directory page renders and the non-mutating UI controls work in an authenticated browser session: data loads, search/filter state updates, tabs switch, and invite/edit modals open and close without console errors. The page is not ready to mark complete because several admin lifecycle actions are not actually end-to-end complete or are under-protected on the backend.

Highest-risk findings:

- Admin invite emails point to `/auth/admin/accept-invite`, but no backend route exists for accepting invitations.
- The invite role dropdown is fed by `/api/admin/roles`, but the backend invite handler rejects the `admin`/`super_admin` roles that the dropdown can present.
- Sensitive account actions such as suspend/activate, session revocation, and force-password-reset only require `AdminUser`, not `admins.manage` or `super_admin`, and some allow self-targeting.

## Frontend Findings

### P1 - Invite Role Dropdown Offers Roles the Invite API Rejects

Location:

- Template: `frontend/platform/admin/admins.html`
- JS: `frontend/platform/static/js/admin-directory.js:176-188`, `frontend/platform/static/js/admin-directory.js:414-447`
- Backend: `backend/src/admin/access.rs:184-210`

Problem:

The frontend populates the invite role dropdown from `/api/admin/roles`, which can return `admin` and `super_admin`. It then defaults to the first returned role. The backend invite handler rejects any role not in `ASSIGNABLE_ROLES`, and `ASSIGNABLE_ROLES` excludes `admin` and `super_admin`. The later elevated-role guard is unreachable for those roles because the allowlist check fails first.

Expected:

The invite dropdown should only show roles the current actor can assign, and the backend should either support super-admin assignment of elevated roles or never expose those roles in this flow.

Evidence:

Browser snapshot showed the invite modal opens. Static review shows `sendInvite()` posts `{ email, role }` from the select. Backend rejects `admin` before checking the super-admin branch.

Recommended fix:

Add a backend endpoint for assignable roles or filter `/api/admin/roles` client-side based on an explicit `assignable` flag. Change backend validation so elevated roles are checked by privilege, not rejected before the elevated-role branch.

### P2 - Unauthenticated Page Load Returns JSON Instead of Login Redirect or Admin Error Page

Location:

- Backend: `backend/src/admin/extractors.rs:133-141`, `backend/src/admin/mod.rs`

Problem:

`GET /admin/admins` is an HTML page route, but unauthenticated access returns `401 {"error":"Authentication required"}` because the page handler uses API-style admin extraction/error behavior.

Expected:

Unauthenticated browser requests to admin pages should redirect to `/auth/login` or render a branded access-denied page, while JSON APIs can keep JSON errors.

Evidence:

`curl -I http://localhost:8888/admin/admins` returned `401 Unauthorized` with `content-type: application/json`.

Recommended fix:

Split HTML page auth rejection from JSON API rejection, or make `page_admin_generic` perform an HTML-friendly redirect/error response.

### P3 - Search Includes Raw UUID but Placeholder Only Mentions Email or Role

Location:

- JS: `frontend/platform/static/js/admin-directory.js:223-231`
- Template: `frontend/platform/admin/admins.html`

Problem:

Search filters by email, names, user id, and role, but the placeholder says only "Search by email or role..."

Expected:

Placeholder should match supported search fields.

Evidence:

Static review of search predicate.

Recommended fix:

Change placeholder to "Search by email, name, ID, or role..."

## Backend Findings

### P1 - Admin Invitations Cannot Be Accepted End-to-End

Location:

- Backend: `backend/src/admin/access.rs:243-254`, `backend/src/admin/access.rs:544-554`
- Route map: `backend/src/main.rs`, `backend/src/auth/routes.rs`, `backend/src/admin/mod.rs`

Problem:

The invite and resend handlers email links to `/auth/admin/accept-invite?token=...`, but no route for `/auth/admin/accept-invite` is registered. Creating/resending an invitation can therefore succeed while the invitee has no working acceptance flow.

Expected:

A registered acceptance route should verify the hashed token, enforce expiry/status, create or link the user, assign the role in a transaction, set `accepted_at`/status, and require password/2FA setup.

Evidence:

`rg "accept-invite"` only found email body strings in `backend/src/admin/access.rs`; no registered route or handler exists.

Recommended fix:

Implement and register an admin-invite acceptance flow, or remove invite/resend UI until the lifecycle exists.

### P1 - Status Updates Are Not Protected by Fine-Grained Admin Permission and Do Not Revoke Sessions

Location:

- JS: `frontend/platform/static/js/admin-directory.js:605-621`
- Backend: `backend/src/admin/users.rs:931-974`

Problem:

The frontend confirmation says suspending an admin revokes active sessions immediately, but the backend status handler only updates `users.status` and writes an audit record. It does not delete sessions, does not block self-targeting, and does not require `admins.manage` or `super_admin`.

Expected:

Suspending an admin should be authorized with `admins.manage` or stricter super-admin rules, block self-suspension unless explicitly intended, prevent demotion/suspension of protected super-admins by lower roles, and revoke sessions in the same transaction.

Evidence:

Handler accepts any `AdminUser` and executes `UPDATE users SET status = $1` without session deletion.

Recommended fix:

Add permission and self/protected-role checks; wrap status update, session revocation, and audit log in one transaction.

### P1 - Session Revocation Is Available to Any AdminUser Without Fine-Grained Permission or Self/Protected-User Guard

Location:

- JS: `frontend/platform/static/js/admin-directory.js:636-647`
- Backend: `backend/src/admin/users.rs:1078-1114`

Problem:

`DELETE /api/admin/users/:user_id/sessions` deletes all sessions for any target user as long as the caller passes `AdminUser`. There is no `admins.manage` check, no self-action guard, and no protected super-admin target rule.

Expected:

Session revocation for admin accounts should require `admins.manage` or super-admin, block accidental self-lockout unless specifically confirmed by backend rules, and enforce target hierarchy.

Evidence:

Handler takes `_admin: AdminUser` and deletes from `user_sessions` directly.

Recommended fix:

Add `require_permission("admins.manage")`, self/protected-role checks, and transactional audit logging.

### P1 - Force Password Reset Swallows Database Errors and Lacks Fine-Grained Permission

Location:

- JS: `frontend/platform/static/js/admin-directory.js:662-673`
- Backend: `backend/src/admin/users.rs:1116-1182`

Problem:

The handler starts a transaction, then uses `let _ =` for the `user_settings` upsert, session deletion, and audit insert. If any of those statements fail, the transaction may still commit and return success. The route also only requires `AdminUser`, not `admins.manage`, and has no self/protected-target guard.

Expected:

Each DB statement should propagate errors and roll back. The action should require `admins.manage` or super-admin and enforce target hierarchy.

Evidence:

`let _ = sqlx::query(...).execute(&mut *tx).await;` is used for all three core writes before commit.

Recommended fix:

Replace ignored results with `?`/`map_err`; add permission and target checks; keep flag update, session revocation, and audit logging in one transaction.

### P2 - Invitation Revoke/Resend Do Not Audit Changes and Resend Partially Ignores Write/Email Failures

Location:

- Backend: `backend/src/admin/access.rs:485-560`

Problem:

Revoking an invitation physically deletes the row and does not write an audit log. Resend updates `expires_at` and `token_hash` with ignored results and ignores email send failure.

Expected:

Invitation changes should be audit logged. Revoke should preferably mark `status='revoked'` so the admin lifecycle is traceable. Resend should fail if token update fails and should return a clear email-delivery status.

Evidence:

`api_admin_invitation_revoke` performs `DELETE FROM admin_invitations`; `api_admin_invitation_resend` uses ignored update/email results.

Recommended fix:

Use status transitions, propagate DB errors, audit old/new state, and expose email send result.

### P2 - Admin List Exposes Admin PII Without Read Audit

Location:

- Backend: `backend/src/admin/access.rs:9-87`

Problem:

The list endpoint returns admin emails, names, last IPs, last activity, session counts, and 2FA status but does not write an audit log. Comparable broader user list code audits PII reads.

Expected:

Admin directory reads should be audit logged because the page exposes security posture and IP metadata for staff accounts.

Evidence:

`api_admin_list` performs the SELECT and returns JSON without an `audit_logs` insert.

Recommended fix:

Insert an `admin.pii_access` or `admin.admin_directory_read` audit record, ideally rate-limited or sampled if needed.

## Security Findings

- P1: Admin invite acceptance route is missing, leaving a security-sensitive lifecycle incomplete after a successful invite.
- P1: Admin status, session revocation, and force-reset endpoints are under-authorized relative to the sensitivity of the actions.
- P1: Force-reset route can report success despite ignored database write failures.
- P2: Invitation revoke/resend lack full audit logging and state transition history.
- P2: Admin directory staff PII/security posture reads are not audit logged.
- CSRF middleware rejected unsafe POST without token during runtime test. `admin-permission-guard.js` is loaded before the page controller and should inject tokens for real browser fetches.
- No monetary logic is present on this page.

---

## Database Findings

- Required tables exist locally: `users`, `user_profiles`, `user_sessions`, `user_settings`, `roles`, `user_roles`, `admin_permissions`, `admin_invitations`, `audit_logs`.
- `admin_invitations` has columns needed to support invite lifecycle (`token_hash`, `status`, `expires_at`, `accepted_at`), but the current revoke handler deletes rows instead of transitioning `status`.
- `user_settings.force_password_reset` and `user_settings.totp_enabled` exist.
- Role permissions exist for `super_admin` and `admin` via `all`; granular `admins.manage` exists for `super_admin`.
- No database transaction is used for status update plus session revocation because the backend does not currently revoke sessions on suspend.
- Local startup logs reported existing reconciliation/token mismatches unrelated to this page audit; they may affect broader admin trust in displayed platform data.

## Missing Tests

- Backend integration test for complete admin invite lifecycle: create invite, accept token, assign role, mark accepted, require password/2FA setup.
- Backend authorization tests for admin status update, session revocation, force reset, role update, invite, resend, and revoke.
- Backend tests that non-super admins cannot mutate super-admin accounts or themselves.
- Backend test that suspend revokes sessions atomically.
- Backend test that force-password-reset rolls back and returns an error when any DB write fails.
- Frontend E2E test that invite role options match assignable backend roles.
- Frontend E2E test for Admin Directory search, filters, tabs, invite modal, edit modal, and API error toasts.

## Recommended Fix Order

1. Implement or disable the admin invite acceptance flow; do not allow successful invite creation when the emailed link is unregistered.
2. Fix role assignment/invite role semantics so UI options and backend allowlists match.
3. Add `admins.manage`/super-admin and self/protected-target guards to status, session revoke, and force-reset endpoints.
4. Make suspend revoke sessions atomically and make force-reset propagate DB errors.
5. Replace invitation hard delete with audited status transitions and audit admin directory reads.
6. Improve unauthenticated HTML page handling and search placeholder copy.
