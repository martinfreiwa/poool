## Frontend Findings

### P2 - Fragment Cards Break Parent Filter Contracts

Location:

- Template/parent JS: `frontend/platform/static/js/marketplace-search.js:86`
- Backend fragment: `backend/src/assets/routes.rs:744`

Problem:

The parent marketplace filters read `card.dataset.location`, `card.dataset.area`, `card.dataset.assetType`, and `card.dataset.duration`, plus selected text selectors such as `[id*="-duration-value"]`. The `/marketplace/tab` fragment emits `.property-card` rows with only `data-property-id`. After a tab swap, dropdown filters cannot reliably filter by location, investment duration, or property type.

Expected:

Every card returned by `/marketplace/tab` should expose the same filter attributes and searchable structure as the initial marketplace grid or a shared card renderer should be used.

Evidence:

`marketplace-search.js` checks `card.dataset.location`, `card.dataset.area`, `card.dataset.assetType`, and `card.dataset.duration`. `api_marketplace_tab` generates `<div class="property-card" data-property-id="{slug}" ...>` only.

Recommended fix:

Render tab fragments from the same card component/data model as the initial `/marketplace` page, including filter data attributes and stable child selectors.

### P2 - Search And Filter Handlers Rebind After Every HTMX Swap

Location:

- JS: `frontend/platform/static/js/marketplace-search.js:377`
- JS: `frontend/platform/static/js/marketplace.js:22`

Problem:

`initMarketplaceSearch()` and `initializeStatusTabs()` are called on DOM ready and again after every `htmx:afterSwap`, but there is no idempotency guard on the persistent parent controls. Each tab switch can add another `input`, `keydown`, `click`, `change`, and document-level `dropdown:change` listener.

Expected:

Persistent marketplace controls should be initialized once, or existing listeners should be replaced/delegated safely. Fragment swaps should only bind behavior to newly inserted fragment nodes.

Evidence:

The HTMX target is `#marketplace-content-wrapper`, so the status/filter controls remain in the DOM while the scripts reattach listeners from document lookups.

Recommended fix:

Add dataset-based initialization guards, event delegation, or AbortController cleanup for search/filter/status handlers.

### P3 - Fragment Card Navigation Is Click-Only

Location:

- Backend fragment: `backend/src/assets/routes.rs:745`

Problem:

The generated card is a `<div>` with an inline `onclick`. It has no native link, `tabindex`, role, or keyboard activation path. Pointer users can open the property detail page, but keyboard users cannot focus and activate the card as a link.

Expected:

Property cards should expose a native `<a href="/property/{slug}">` or equivalent accessible keyboard behavior with clear focus styling.

Evidence:

The fragment emits `<div class="property-card" data-property-id="{slug}" onclick="window.location.href='/property/{slug}'">`.

Recommended fix:

Use a real anchor for the card target, or add `tabindex="0"`, role, Enter/Space handling, and visible focus state while preserving gallery button behavior.

## Backend Findings

### P1 - Database Failures Are Masked As Empty Marketplace Tabs

Location:

- Backend: `backend/src/assets/routes.rs:628`

Problem:

The fragment query ends with `.fetch_all(&state.db).await.unwrap_or_default()`. If PostgreSQL is down, the query shape breaks, or a migration introduces a mismatch, the user sees "No properties found in this category" with an HTTP 200 instead of an operational error.

Expected:

DB failures should be logged and surfaced as a safe error state/status so the frontend can show retry messaging and operators can detect the failure.

Evidence:

The handler returns the same empty-state branch for both true zero results and any query error.

Recommended fix:

Replace `unwrap_or_default()` with explicit error handling, return a non-2xx response or error fragment, and add an E2E or integration test for DB-failure behavior.

### P2 - Fragment Markup Diverges From Initial Marketplace Card Semantics

Location:

- Initial card: `frontend/platform/index.html:302`
- Fragment card: `backend/src/assets/routes.rs:744`

Problem:

The initial page renders a funding-status badge, token price, annual yield, and capital appreciation rows. The fragment renders a lease-type badge, investment duration, projected return, and projected annualised return. Switching tabs changes the card information architecture for the same marketplace surface.

Expected:

Available/funded tab swaps should preserve the same card schema unless the product intentionally defines different card variants.

Evidence:

The initial card badge text is funding status while the fragment badge text is lease label; the row labels also differ.

Recommended fix:

Consolidate card rendering through one MiniJinja partial or one backend helper with the same fields for initial and HTMX responses.

## Security Findings

- No auth bypass found: `api_marketplace_tab` checks `get_current_user` and returns 401 for missing session.
- HTML escaping is present for DB-backed string fields used in the manually-built fragment.
- No CSRF concern for this endpoint because it is read-only.
- No sensitive data exposure found in the selected query; it returns public asset data for authenticated users.
- Operational security/reliability concern: DB failures are masked as successful empty fragments, which can hide marketplace outages.

## Database Findings

- Required tables exist in schema docs and migrations: `assets`, `asset_images`.
- Monetary fields used by the fragment are integer cents (`total_value_cents`, `token_price_cents`), but the fragment displays only total value and does not expose token price like the initial grid.
- The query filters by `published = true`, `funding_status`, and `asset_type != 'commodity'`, matching the parent property marketplace scope.
- No database writes occur.
- Missing error propagation is the main data-integrity/operability issue.
