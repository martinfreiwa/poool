## Summary

`/auth/2fa` is the post-password TOTP verification page. The core happy path is implemented: login creates a session with `is_2fa_verified = false`, the page accepts a 6-digit TOTP code, the backend verifies it against `user_settings.totp_secret`, rotates the session token, marks the session 2FA-verified, rotates CSRF, and redirects to `/marketplace`.

2026-04-28 fix pass: the documented page issues are fixed locally. `POST /auth/2fa` now uses dedicated `2fa:ip` and `2fa:user` auth-rate-limit buckets, invalid-code/rate-limit responses return HTMX-safe HTML error fragments, the page no longer depends on the unloaded response-targets extension, and the code input/error/loading states are accessible and duplicate-submit aware.

Final status is `fixed, needs authenticated E2E recheck`.

## Frontend Findings

### P1 - Invalid 2FA errors are unlikely to render

Location:

- Template: `frontend/platform/auth-2fa.html:19-21`
- Backend: `backend/src/auth/routes.rs:321-331`
- Shared head: `frontend/platform/components/head.html:190`

Problem:

The form declares `hx-ext="response-targets"` and `hx-target-400="#auth-error"`, but the response-targets extension is not loaded anywhere in the reviewed sources. On invalid TOTP, the handler returns `AppError::Unauthorized`, which becomes a non-2xx JSON response. HTMX does not normally swap non-2xx responses unless configured. Unlike `login.html`, this page does not add a local `htmx:beforeSwap` handler to force error swaps.

Impact:

Users entering an incorrect code, expired session, or missing/invalid CSRF token can be left on the page without a visible inline error. This is especially risky during login recovery because users may retry blindly or abandon the flow.

Expected:

Use a local or shared HTMX error swap handler, return a page-friendly HTML error fragment for HTMX submissions, or load the actual response-targets extension from a trusted/local asset. Keep error output inside `#auth-error` with `role="alert"` or `aria-live`.

### P1 - TOTP verification has no dedicated attempt/rate limit

Location:

- Backend: `backend/src/auth/routes.rs:300-348`
- Service: `backend/src/auth/service.rs:864-910`

Problem:

`POST /auth/login` has IP and email rate limiting, but `POST /auth/2fa` does not call the shared auth rate limiter or maintain a per-session/per-user attempt counter. After an attacker has a valid password and unverified session, they can repeatedly submit TOTP guesses until the session expires. Redis replay protection only blocks reuse of an already valid code; it does not throttle invalid guesses.

Impact:

This weakens the second factor against online guessing, especially for compromised passwords. Six-digit TOTP space is small enough that a post-password throttle is mandatory.

Expected:

Add a rate limit keyed by user/session and client IP, for example `2fa:{user_id}` and `2fa:ip:{client_ip}`, with lockout/backoff and audit/Sentry warning on repeated failures. Add tests for invalid-code throttling and reset behavior after success.

### P2 - 2FA form accessibility is incomplete

Location:

- Template: `frontend/platform/auth-2fa.html:19-29`

Problem:

The label points at `code-input`, but the input is `id="email-input"`. The error container has no `aria-live` or `role="alert"`, so screen-reader users may not hear errors. The input also lacks `autocomplete="one-time-code"`, which would help mobile and password-manager flows.

Impact:

The key login security step is harder to complete with assistive technology and mobile autofill.

Expected:

Rename the input to `id="code-input"` or update the label, add `aria-live="polite"`/`role="alert"` to the error area, and add `autocomplete="one-time-code"`.

### P2 - Submit/loading state is not visible or duplicate-submit safe

Location:

- Template: `frontend/platform/auth-2fa.html:20-40`

Problem:

The form references `hx-indicator="#loading-indicator"`, but no `#loading-indicator` exists. The submit button is not disabled during request, and there is no pending text such as "Verifying...". Duplicate submits are possible in the UI; replay protection helps only when Redis is healthy and only for a valid code.

Impact:

Users get weak feedback during a sensitive login step, and accidental double submits can create confusing failures.

Expected:

Add a real loading indicator/status region, disable the submit button during request, restore it on failure, and keep the replay guard as backend defense-in-depth.

## Backend Findings

### Working behavior

- `POST /auth/login` creates a session with `is_2fa_verified = false` when `user_settings.totp_enabled` is true.
- `GET /auth/2fa` uses `get_user_by_session_unverified`, which allows the intermediate unverified session to render the page.
- `POST /auth/2fa` reads `user_settings.totp_secret`, verifies the code, rotates the session token, sets `is_2fa_verified = true`, rotates CSRF, and redirects with `HX-Redirect: /marketplace`.
- `get_user_by_session` denies normal authenticated access when TOTP is enabled but the session is not verified.
- TOTP replay protection exists when Redis is configured.

### Risks and follow-up checks

- `POST /auth/2fa` needs dedicated rate limiting for invalid codes.
- Redis replay guard degrades open when Redis is missing or unavailable. That may be acceptable for availability, but it increases the need for DB/session-level attempt limiting.
- Direct expired/no-session visits return an `AppError::Unauthorized` JSON body rather than redirecting to `/auth/login`; this is acceptable for security, but poor UX and should be considered during the fix.
- There is no committed route/UI test that proves invalid code feedback, CSRF behavior, token rotation, or attempt throttling for this page.
