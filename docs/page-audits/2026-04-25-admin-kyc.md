## Summary

`/admin/kyc` has a real SSR page, a KYC table controller, and backend list/document/approve/reject routes. The unauthenticated page/API boundary correctly returns `401`, `admin-kyc.js` parses, and `cargo check` passes.

The page is not production-ready for KYC/AML operations. KYC-specific permissions are not enforced on the page or API handlers, sensitive document access can succeed without durable audit guarantees, approval/rejection mutations do not write KYC decision audit records, and several backend failures are rendered as empty UI states. Authenticated browser mutation testing was not performed because no admin test session was available in this audit run.

## Frontend Findings

### P2 - Document viewer injects signed URL data with raw HTML

Location:

- JS: `frontend/platform/static/js/admin-kyc.js:520`

Problem:

`viewKYCDocuments()` builds `<embed>` and `<img>` markup with `content.innerHTML = docs.map(...)`, placing backend-returned signed URLs directly into HTML attributes.

Expected:

Render document previews with DOM APIs (`createElement`, `setAttribute`) after validating each URL is an expected HTTPS signed-storage URL. Use `textContent` for footer text.

Evidence:

The code maps `d.url` into `src="${d.url}"`; KYC document metadata originates from private uploads and the signed URL generation path. Even if the backend usually returns a trusted GCS URL, this page handles sensitive identity documents and should avoid attribute injection patterns.

Recommended fix:

Replace the string template with DOM node creation, validate URL protocol/host, and render a user-visible error for invalid or missing URLs.

### P2 - Modal and table controls have incomplete accessibility behavior

Location:

- Template: `frontend/platform/admin/kyc.html:294`
- JS: `frontend/platform/static/js/admin-kyc.js:349`

Problem:

The KYC review modal and document viewer do not set `role="dialog"`, `aria-modal`, focus management, Escape handling, or focus restoration. Sortable table headers are clickable `<th>` elements but are not keyboard buttons and do not expose sort state.

Expected:

Dialogs should trap focus, close on Escape, restore focus to the invoking button, and expose accessible names. Sort controls should be keyboard-accessible buttons or have equivalent semantics.

Evidence:

`setupModal()` only handles cancel click and backdrop click. The document viewer close control is an inline clickable `<div>`.

Recommended fix:

Apply the accessible modal pattern already used in newer admin pages and convert sortable headers or their contents into semantic buttons with `aria-sort`.

## Backend Findings

### P1 - KYC page and APIs do not enforce KYC-specific permissions

Location:

- Backend page route: `backend/src/admin/pages.rs:140`
- API routes: `backend/src/admin/kyc.rs:13`, `backend/src/admin/kyc.rs:93`, `backend/src/admin/kyc.rs:144`, `backend/src/admin/kyc.rs:189`

Problem:

The page is served by generic admin routing and the API handlers only require `AdminUser`. The repo defines `kyc.read`, `kyc.write`, `kyc.override`, and `pii.view`, and the sidebar maps KYC navigation to `kyc.read`, but the backend does not enforce those permissions here. The `AdminUser` extractor also only accepts `admin` and `super_admin`, so a compliance role with KYC permissions is not enough unless that user also has an admin role.

Expected:

`GET /admin/kyc` and read/list routes should require `kyc.read`; document URLs should require `kyc.read` plus `pii.view`; approve/reject should require `kyc.write` or `kyc.override` according to the decision model. Compliance users with the appropriate permission should be supported by the backend access model if the seeded roles intend that.

Evidence:

`backend/src/admin/extractors.rs` checks only `r.name IN ('admin', 'super_admin')`; no `require_permission()` call appears in `backend/src/admin/kyc.rs`.

Recommended fix:

Add explicit permission checks to the page and each KYC API, and decide whether `AdminUser` should become a staff/admin extractor that allows non-admin compliance/support roles with scoped permissions.

### P1 - KYC decisions lack durable audit logs and atomic side effects

Location:

- Backend: `backend/src/admin/kyc.rs:144`

Problem:

Approving or rejecting a KYC record updates `kyc_records` but does not write an `audit_logs` decision record. Approval then updates `affiliate_referrals` in a best-effort second write and ignores errors, so the response can report success even if the referral state machine was not advanced.

Expected:

KYC approve/reject should run in a transaction, lock or conditionally update the target record, persist previous/new state to `audit_logs`, and atomically apply dependent referral state changes or report failure.

Evidence:

The approve handler performs `UPDATE kyc_records`, then `let _ = sqlx::query(... UPDATE affiliate_referrals ...)`. The reject handler only updates `kyc_records`.

Recommended fix:

Wrap each decision in a `sqlx::Transaction`, capture the previous KYC status/reason, write an audit log with actor/admin ID, and propagate dependent write failures.

### P1 - KYC document signed URL access is overbroad and audit logging is best effort

Location:

- Backend: `backend/src/admin/kyc.rs:93`

Problem:

`GET /api/admin/kyc/:kyc_id/documents` returns signed URLs for identity documents with only broad admin access. The audit insert is `let _ = ...`, so a failed audit write does not block signed URL issuance.

Expected:

Sensitive identity-document access should require explicit KYC/PII permission and should not mint signed URLs if audit logging fails, unless there is a documented break-glass policy.

Evidence:

The handler inserts into `audit_logs` but ignores the result before selecting documents and generating signed URLs.

Recommended fix:

Require `kyc.read` and `pii.view`, propagate audit write failures, include actor/IP/user-agent metadata where available, and consider a shorter signed URL TTL for admin preview.

### P2 - Backend failures are converted into empty UI states

Location:

- Backend: `backend/src/admin/kyc.rs:60`, `backend/src/admin/kyc.rs:116`, `backend/src/admin/kyc.rs:132`
- JS: `frontend/platform/static/js/admin-kyc.js:108`

Problem:

KYC list and document queries use `unwrap_or_default()`. Signed URL generation also falls back to an empty string. The frontend does not render a visible error when the list API returns non-2xx.

Expected:

Database and storage failures should return safe 5xx errors and the page should show a retryable error state. Empty queue should only mean a successful empty response.

Evidence:

`api_admin_kyc_records()` uses `.fetch_all(...).await.unwrap_or_default()`. `api_admin_kyc_documents()` does the same for documents and URL generation.

Recommended fix:

Map DB/storage errors into `ApiError`, log server-side detail, and add visible load/document error states in the UI.

### P2 - Rejection reason is only enforced client-side

Location:

- JS: `frontend/platform/static/js/admin-kyc.js:378`
- Backend: `backend/src/admin/kyc.rs:189`

Problem:

The frontend requires a rejection reason, but the backend accepts a missing reason and stores `"No reason provided"`. It also does not enforce length or structured reason limits.

Expected:

The backend should reject blank rejection reasons and enforce reasonable length/allowed content constraints because KYC rejection reasons are compliance records.

Evidence:

The reject handler calls `.unwrap_or("No reason provided")`.

Recommended fix:

Deserialize a typed request body and validate `rejection_reason.trim()` server-side with a maximum length.

- `audit_logs` exists but KYC decision handlers do not use it.
- `kyc.read`, `kyc.write`, `kyc.override`, and `pii.view` permissions exist in migrations, but the KYC handlers do not check them.
- Approval side effects on `affiliate_referrals` are not transactional with the KYC approval.

## Recommended Fix Order

1. Enforce KYC/PII permissions on page, list, document, approve, and reject routes.
2. Make approve/reject transactional and audit logged; propagate affiliate-referral side-effect errors.
3. Make document signed URL access fail closed if audit logging or storage signing fails.
4. Add backend validation for rejection reasons and visible frontend API error states.
5. Replace document viewer `innerHTML` rendering and add accessible modal/focus behavior.
6. Add targeted backend and E2E regression tests.
